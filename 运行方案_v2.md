# Timing ECO Agent Team 整体运行方案（改进版）

> 芯片物理设计中的时序 ECO 修复多 Agent 系统
> 版本：v2（基于 CEO Review 改进）
> 改进日期：2026-04-02

---

## 一、系统概述

### 1.1 背景

在芯片物理设计中，时序修复是一个迭代过程，需要反复执行：寄生参数提取 → 静态时序分析 → 分析报告 → 执行修复策略 → 验证。每次迭代可能涉及数百甚至数千条时序违规，人工分析效率低且容易遗漏。

### 1.2 解决方案

通过多 Agent 协作系统，模拟人类工程师的工作流程：
- 每个 Agent 专注于特定任务
- Agent 间通过结构化消息传递上下文
- 质量门确保每个阶段输出符合标准
- **Log Monitor 作为 Orchestrator 的内置功能**实时监控日志，确保异常及时发现并终止
- 迭代循环支持持续优化直到签核完成

### 2.2 Log Monitor（Orchestrator 内置功能）

Log Monitor 不作为独立 Agent，而是集成在 Orchestrator 内部，实时解析 EDA 工具输出日志。

#### 职责范围

- 监控所有 EDA 工具运行日志（StarRC、PTEco、PrimeTime、Innovus）
- 解析 Agent 执行日志
- 监控系统资源使用日志

#### 告警级别定义

| 级别 | 名称 | 说明 | 处理方式 |
|------|------|------|---------|
| LEVEL 1 | INFO | 正常进度更新 | 记录，不告警 |
| LEVEL 2 | WARNING | 可能存在问题，需人工审查 | 继续执行，在报告中标记 |
| LEVEL 3 | ERROR | 执行失败或异常 | 尝试恢复或回滚 |
| LEVEL 4 | FATAL | 无法继续执行 | **立即终止**，通知 Orchestrator |

#### FATAL 错误判断标准

```
TERMINATE_IF:
  1. 工具内部错误 (Internal Error, Core Dump)
  2. 数据损坏 (Checksum mismatch, File corrupted)
  3. 关键资源失败 (License server down, Disk full)
  4. 设计数据无效 (Floating nets, Orphan pins)
  5. 内存耗尽 (OOM, Allocation failed)
  6. 致命语法错误 (Parse error in critical file)

NOT_TERMINATE_IF:
  1. 非关键警告 (Info messages, Progress updates)
  2. 可恢复错误 (Retry succeeded, Fallback used)
  3. 阈值警告 (Margin below target, but positive)
  4. 建议性消息 (Suggestions, Recommendations)
```

### 3.2 消息类型定义

| 消息类型 | 说明 | 方向 | 携带内容 |
|---------|------|------|---------|
| `TASK_REQUEST` | 任务请求 | Orchestrator → Agent | 任务描述、输入数据、执行指令 |
| `TASK_RESPONSE` | 任务响应 | Agent → Orchestrator | 执行结果、输出数据、状态 |
| `HANDOVER` | 交接消息 | Agent → Agent | 完整上下文、前一阶段输出 |
| `QUALITY_GATE` | 质量门 | Agent → Orchestrator | 验证结果、GO/NO-GO 决策 |

| `TERMINATE` | 终止信号 | Orchestrator(Log Monitor) → All | 终止原因、错误详情 |
| `ITERATION` | 迭代信号 | Orchestrator → Agent | 迭代号、返回阶段 |
| `ESCALATE` | 升级信号 | Orchestrator → Human | 升级原因、完整状态、建议 |

#### 5.1.2 Orchestrator 核心逻辑

```python
class Orchestrator:
    def run_pipeline(self, workflow_type):
        iteration = 0
        while iteration < MAX_ITERATIONS:
            # === 阶段执行 ===
            phase_output = self.execute_current_phase()

            # === 质量门检查 ===
            gate_result = self.check_quality_gate(phase_output)
            if gate_result == "PASS":
                if is_final_phase():
                    return "GO"
                self.advance_to_next_phase()
            elif gate_result == "CONDITIONAL":
                self.proceed_with_monitoring()
            else:  # NO-GO
                # === 新增：校准检查（Calibration Check）===
                if not self.calibration_check():
                    self.escalate_to_human()  # 精确升级报告
                    return "ESCALATE"

                self.return_to_phase(phase_to_return_to)
            iteration += 1

        self.escalate_to_human()  # 超过最大迭代次数
        return "ESCALATE"

    def calibration_check(self):
        """NO-GO 后，验证违规是否仍然真实存在"""
        # 1. 从最新时序报告中提取 WNS/TNS
        # 2. 与 ViolationAnalyzer 报告的违规列表对比
        # 3. 如果违规数量减少 >50%，说明 ECO 有效果，继续迭代
        # 4. 如果违规数量几乎不变，说明根因判断可能有误
        # 5. 返回 True = 继续迭代，False = 升级人工
```

#### 5.1.3 EDA 工具调用方式

| 工具 | 调用方式 | 说明 |
|------|---------|------|
| StarRC | `subprocess.run([...])` | 批处理模式，输出到指定目录 |
| PTEco | `subprocess.run([...])` 或 API | 优先使用命令行，API 备选 |
| PrimeTime | `subprocess.run([...])` | PT shell 批处理模式 |
| Innovus | `subprocess.run([...])` | Innovus batch mode |

#### 5.1.4 Agent 执行模型

每个 Agent 是一个 prompt 模板 + LLM API 调用：

1. **加载**：从 `agents/{agent_name}.md` 读取 prompt 模板
2. **注入上下文**：将上一阶段输出填入 prompt 的对应位置
3. **LLM 调用**：使用配置的 LLM（Claude/GPT 等）生成输出
4. **解析验证**：使用输出 schema 验证格式，验证失败则重试（最多3次）
5. **消息发送**：将输出打包为 HANDOVER 消息，传递给下一 Agent

---

### 5.2 质量门阈值

| 阶段 | 质量门标准 | 失败处理 |
|------|-----------|----------|
| 数据解析 | 数据完整，格式正确，日志无 FATAL 错误 | 重试（最多 3 次） |
| 违规分析 | >80% 违规已分类，置信度 >70% | 重试或升级 |
| 策略生成 | 每个违规簇 ≥2 个策略 | 重试或升级 |
| **根因验证（新增）** | 根因与原始时序路径特征一致，置信度 >75% | 返回阶段 2 重新分析 |
| ECO 执行 | 无执行错误，DRC 通过，日志无 ERROR | 回滚并重试 |
| 签核验证 | WNS ≥ 0, TNS = 0 | 迭代循环 |

---

### 5.3 迭代限制

- **单阶段重试次数**：最多 3 次
- **总迭代次数**：最多 10 次
- **超过限制**：精确升级报告 → 人工干预

## 六、决策逻辑

### 6.1 质量门决策

```
如果 WNS >= 0 AND TNS == 0 AND 所有 margin 达标：
    → GO：流程完成

如果 WNS >= 0 但 margin < 阈值：
    → CONDITIONAL：带监控继续

如果 WNS < 0：
    → NO-GO：执行 Calibration Check
        → 如果违规仍然存在：继续迭代，返回阶段 2 或 3
        → 如果违规已消失：升级人工（可能工具报告不一致）

如果阶段 4 执行错误：
    → NO-GO：重试实施，返回阶段 4 修正命令

如果日志中出现 FATAL 错误：
    → TERMINATE：立即终止当前操作
    → ESCALATE：发送精确升级报告

如果超过最大迭代次数（10）：
    → ESCALATE：需要人工干预
```

### 6.2 根因验证（Cross-Validate）决策

```
ECOStrategyGenerator 报告根因: [LOGIC_DEPTH]
       │
       ▼
从 ViolationAnalyzer 获取原始时序路径特征:
  - path_depth: 8 logic levels
  - cell_types: [UCB4, NAND2x4, ...]
  - net_cap: 0.32 pF
       │
       ▼
根因与路径特征对比:
  - LOGIC_DEPTH 要求: path_depth > 6
  - 实际: path_depth = 8  ✓ 一致
  - 预期 ECO: 逻辑优化减少 2 个逻辑层级
       │
       ▼
如果一致:
    → VALIDATED：输出 ECO 策略给 CadenceToolIntegrator
如果不一致:
    → FLAG_FOR_REVIEW：标记该策略需人工审核
    → 记录不一致原因到迭代历史
    → 继续处理其他已验证的策略
```

## 九、测试与验证策略（新增）

### 9.1 测试分层

| 测试类型 | 说明 | 覆盖内容 |
|---------|------|---------|
| **Agent 单元测试** | 对每个 Agent prompt 输出格式和逻辑进行独立测试 | 输出 schema 验证、根因分类准确性 |
| **Pipeline 集成测试** | 端到端运行完整流程，验证 agent 间 handoff | 上下文完整性、质量门决策、迭代收敛 |
| **Golden Dataset 测试** | 使用已知结果的时序数据验证系统输出 | WNS/TNS 改善值、违规分类准确性 |
| **故障注入测试** | 模拟 FATAL 错误、日志异常、数据缺失 | 错误检测、终止逻辑、升级报告 |

### 9.2 Golden Dataset

准备 3-5 个已知结果的时序案例：

| Dataset | 特点 | 预期结果 |
|---------|------|---------|
| case_1.csv | 100条违规，LOGIC_DEPTH 主导 | 系统应识别 LOGIC_DEPTH 簇并建议逻辑优化 |
| case_2.csv | 50条违规，CELL_SIZING 主导 | 系统应识别 CELL_SIZING 并建议 upsizing |
| case_3.csv | 空违规集 | 系统应输出 GO 而非报错 |
| case_4.csv | 数据缺失（缺少 SPEF） | 系统应检测到数据不完整并升级 |
| case_5.csv | StarRC FATAL 错误注入 | 系统应检测 FATAL 并终止 |

### 9.3 验收标准

系统在 golden dataset 上的验收标准：

- **准确率**：>85% 的违规根因分类与专家判断一致
- **召回率**：>90% 的真实违规被识别（无遗漏）
- **收敛率**：>80% 的测试案例在 ≤10 次迭代内达到 GO 或 CONDITIONAL
- **误报率**：Cross-Validate 阶段，根因不一致标记率 <15%
- **升级报告质量**：人工审查升级报告后，判断"选项 A"（继续迭代）正确的比例 >70%

## 十、使用建议

### 10.1 何时使用哪个工作流

| 场景 | 推荐工作流 |
|------|-----------|
| 首次时序修复 | 工作流一：时序 ECO 签核 |
| ECO 迭代后验证 | 工作流二：时序对比 |
| 剩余 < 50 条违规 | 工作流三：最终清理 |
| 版本/阶段对比 | 工作流二：时序对比 |

### 10.2 成功关键

1. **上下文完整性**：每个 Agent 的输出必须完整传递给下一个 Agent
2. **日志监控及时性**：FATAL 错误立即终止，避免无效计算
3. **质量门严格性**：不达标不推进，避免问题积累
4. **根因验证（Cross-Validate）**：防止错误根因的 ECO 策略被执行
5. **校准检查**：NO-GO 后确认违规是否仍然存在，避免无效迭代
6. **迭代控制**：及时识别不收敛情况，升级人工干预

### 10.3 常见问题处理

| 问题 | 处理方式 |
|------|---------|
| Parser 无法解析文件 | 检查文件格式，确认工具版本兼容性 |
| StarRC 内存不足 | FATAL 终止，升级人工，增加资源或分区处理 |
| PTEco 工具错误 | FATAL 终止，记录日志，升级人工 |
| 违规分类置信度低 | 返回 ViolationAnalyzer 进行更深入分析 |
| Cross-Validate 失败 | 返回 ViolationAnalyzer 重新分析该簇 |
| 策略执行 DRC 失败 | ERROR 处理，回滚，修正策略 |
| 迭代不收敛 | 评估是否需要人工 ECO，调整策略方向 |
| **Calibration Check 失败** | **升级人工，确认是否有工具报告不一致问题** |
